## 1. Overview

This document captures the functional requirements, API design, database schema, and security architecture for a custom authentication system built with FastAPI. The system enforces a **two-gate approval workflow** specific to CVS corporate email accounts: new users must verify their email address and separately receive approval from a designated administrator before gaining access.

The design prioritizes security through password hashing, JWT-based session tokens, and short-lived one-time verification codes distributed via email.

---

## 2. Functional Requirements

### 2.1 Sign-Up Flow

#### REQ-S01: Email Domain Validation
Only CVS corporate email addresses (ending in `@cvs.com`) are accepted for registration. Any other domain must be rejected with a clear error message at the point of submission.

#### REQ-S02: Duplicate Account Prevention
Before creating a new record, the system must check whether the submitted email address already exists in the database. If a record is found, the API returns a conflict error and no new record is created.

#### REQ-S03: Secure Password Storage
Passwords must never be stored in plain text. Upon successful registration, the password is hashed using **bcrypt** (work factor ≥ 12) before being persisted to the database.

#### REQ-S04: Dual Verification Code Generation
On successful registration, the system generates two independent, cryptographically random verification tokens:

- A **user verification token** sent to the registering user's email.
- An **admin approval token** sent to a pre-configured administrator email address.
- Both tokens must be unique, URL-safe, and carry a defined expiry (recommended: **24 hours**).

#### REQ-S05: Verification Email Content
Each verification email must contain a clickable URL in the format:

```
GET /api/v1/auth/verify/{token}
```

Clicking the URL triggers the corresponding status change for that token type (email verified or admin approved).

#### REQ-S06: Email Verification Status Change
When a user clicks their verification link, the system must:

1. Validate the token exists and has not expired.
2. Update the user's `email_verified` flag to `TRUE`.
3. Invalidate the token so it cannot be reused.

#### REQ-S07: Admin Approval Status Change
When the admin clicks their approval link, the system must:

1. Validate the token exists and has not expired.
2. Update the user's `admin_approved` flag to `TRUE`.
3. Invalidate the token so it cannot be reused.

---

### 2.2 Sign-In Flow

#### REQ-L01: Prerequisite Gating
A user may only sign in if **both** conditions are met:

- Their email address has been verified (`email_verified = TRUE`).
- Their account has been approved by an administrator (`admin_approved = TRUE`).

Any attempt to sign in before both gates are cleared must return an informative `403 Forbidden` response indicating which condition is unmet.

#### REQ-L02: Credential Validation
The system must verify the submitted password against the stored bcrypt hash. An incorrect password returns a `401 Unauthorized` response.

#### REQ-L03: JWT Token Issuance
Upon successful login, the system issues a signed JWT containing:

| Claim | Value |
|---|---|
| `sub` | User's email or UUID |
| `exp` | Short-lived expiry (e.g., 1 hour) |
| `iat` | Timestamp of issuance |
| `role` | User role for authorization (`user` / `admin`) |

#### REQ-L04: Cookie-Based Token Delivery
For browser-based frontends, the JWT access token is set as an `HttpOnly`, `Secure`, `SameSite=Lax` cookie on the response. This prevents JavaScript-based XSS access to the token.

#### REQ-L05: Protected Endpoint Authorization
All endpoints beyond the auth flow require a valid JWT. The token can be supplied either via the HttpOnly cookie (browser clients) or the `Authorization: Bearer` header (API clients). An expired or invalid token returns a `401` response.

## 3. User Account State Machine

A user account progresses through the following states from registration to full access:

| State | `email_verified` | `admin_approved` | Description |
|---|:---:|:---:|---|
| `PENDING` | `false` | `false` | Account just created; no actions taken. |
| `EMAIL_VERIFIED` | `true` | `false` | User clicked their email link; awaiting admin. |
| `ADMIN_APPROVED` | `false` | `true` | Admin approved before user verified email. |
| `ACTIVE` | `true` | `true` | Fully active; sign-in is permitted. |
| `DISABLED` | — | — | Account manually disabled by an admin. |

> **Note:** `EMAIL_VERIFIED` and `ADMIN_APPROVED` are independent states. Either can complete first. Sign-in is only granted when both flags are `TRUE` (the `ACTIVE` state).

```
[Registration]
      │
      ▼
  PENDING ──── user clicks link ────► EMAIL_VERIFIED
      │                                     │
      │                                     │
      └──── admin clicks link ───► ADMIN_APPROVED
                                            │
                             (both true) ───▼
                                         ACTIVE ──► DISABLED
```

## 4. API Endpoint Design

All endpoints are prefixed with `/api/v1`. Auth endpoints live under `/auth`; protected resources are grouped separately.

### 4.1 Public Endpoints (No Auth Required)

---

#### `POST /api/v1/auth/signup`

Register a new CVS user. Validates domain, hashes password, creates user record, generates tokens, and sends verification emails.

**Request Body**
```json
{
  "email": "user@cvs.com",
  "password": "SecurePass123!"
}
```

**Response — 201 Created**
```json
{
  "message": "Registration successful. Please check your email to verify your account."
}
```

**Error Responses**

| Code | Reason |
|---|---|
| `400 Bad Request` | Invalid email domain or password does not meet policy. |
| `409 Conflict` | Email address already registered. |

---

#### `GET /api/v1/auth/verify/{token}`

Unified verification endpoint consumed by both user email links and admin approval links. The token type (`email_verify` vs `admin_approve`) is resolved server-side from the token record.

**URL Parameter:** `token` — the unique random string from the email link.

**Response — 200 OK**
```json
{
  "message": "Email verified successfully." 
}
```

**Error Responses**

| Code | Reason |
|---|---|
| `400 Bad Request` | Token has expired. |
| `404 Not Found` | Token not found or already consumed. |

---

#### `POST /api/v1/auth/signin`

Authenticate a fully-verified and approved user. Returns a signed JWT and sets it as an HttpOnly cookie for browser clients.

**Request Body**
```json
{
  "email": "user@cvs.com",
  "password": "SecurePass123!"
}
```

**Response — 200 OK** *(cookie also set: `access_token`, HttpOnly, Secure, SameSite=Lax)*
```json
{
  "access_token": "<jwt>",
  "token_type": "bearer",
  "expires_in": 3600
}
```

**Error Responses**

| Code | Reason |
|---|---|
| `401 Unauthorized` | Invalid credentials. |
| `403 Forbidden` | Email not verified, or account not yet admin-approved. |

---

#### `POST /api/v1/auth/signout`

Clears the HttpOnly session cookie on the client. The JWT remains cryptographically valid until its `exp` claim (stateless design).

**Response — 200 OK**
```json
{ "message": "Signed out successfully." }
```

---

### 4.2 Protected Endpoints (JWT Required)

All routes below require a valid JWT via cookie or `Authorization: Bearer` header. A FastAPI dependency `get_current_user` handles extraction and validation.

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/v1/users/me` | Return the authenticated user's profile. |
| `PATCH` | `/api/v1/users/me` | Update the authenticated user's profile. |
| `GET` | `/api/v1/admin/users` | **Admin only:** list all registered users. |
| `PATCH` | `/api/v1/admin/users/{id}` | **Admin only:** enable or disable an account. |

## 5. Database Design (PostgreSQL)

The system uses PostgreSQL with two core tables. All timestamps are stored in UTC. UUIDs are used as primary keys to prevent enumeration attacks.

### 5.1 Table: `users`

Stores all registered user accounts and their current verification state.

| Column | Type | Nullable | Default | Description |
|---|---|:---:|---|---|
| `id` | `UUID` | NO | `gen_random_uuid()` | Primary key. |
| `email` | `VARCHAR(254)` | NO | — | CVS email. Unique index enforced at DB level. |
| `hashed_password` | `VARCHAR(72)` | NO | — | bcrypt hash of the user's password. |
| `email_verified` | `BOOLEAN` | NO | `FALSE` | Set to `TRUE` after user clicks their email link. |
| `admin_approved` | `BOOLEAN` | NO | `FALSE` | Set to `TRUE` after admin clicks their approval link. |
| `is_active` | `BOOLEAN` | NO | `TRUE` | `FALSE` disables the account manually. |
| `role` | `VARCHAR(20)` | NO | `'user'` | Role for RBAC: `'user'` or `'admin'`. |
| `created_at` | `TIMESTAMPTZ` | NO | `now()` | Account creation timestamp (UTC). |
| `updated_at` | `TIMESTAMPTZ` | NO | `now()` | Last modification timestamp (UTC). |

**DDL**
```sql
CREATE TABLE users (
    id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    email            VARCHAR(254) NOT NULL UNIQUE,
    hashed_password  VARCHAR(72)  NOT NULL,
    email_verified   BOOLEAN      NOT NULL DEFAULT FALSE,
    admin_approved   BOOLEAN      NOT NULL DEFAULT FALSE,
    is_active        BOOLEAN      NOT NULL DEFAULT TRUE,
    role             VARCHAR(20)  NOT NULL DEFAULT 'user',
    created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE INDEX idx_users_email ON users (email);
```

---

### 5.2 Table: `verification_tokens`

Stores one-time tokens for email verification and admin approval. Each registration creates **two** records: one of type `email_verify` and one of type `admin_approve`.

| Column | Type | Nullable | Default | Description |
|---|---|:---:|---|---|
| `id` | `UUID` | NO | `gen_random_uuid()` | Primary key. |
| `user_id` | `UUID` | NO | — | FK → `users.id`. `ON DELETE CASCADE`. |
| `token` | `VARCHAR(128)` | NO | — | Unique random token (`secrets.token_urlsafe`). |
| `token_type` | `VARCHAR(20)` | NO | — | `'email_verify'` or `'admin_approve'`. |
| `expires_at` | `TIMESTAMPTZ` | NO | — | Token expiry (e.g., `now() + interval '24 hours'`). |
| `used_at` | `TIMESTAMPTZ` | YES | `NULL` | `NULL` until consumed; set on first use. |
| `created_at` | `TIMESTAMPTZ` | NO | `now()` | Token generation timestamp. |

**DDL**
```sql
CREATE TABLE verification_tokens (
    id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token       VARCHAR(128) NOT NULL UNIQUE,
    token_type  VARCHAR(20)  NOT NULL CHECK (token_type IN ('email_verify', 'admin_approve')),
    expires_at  TIMESTAMPTZ  NOT NULL,
    used_at     TIMESTAMPTZ,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE INDEX idx_tokens_token ON verification_tokens (token);
CREATE INDEX idx_tokens_user  ON verification_tokens (user_id);
```

---

### 5.3 Entity Relationship

```
users
  id (PK) ◄────────────────────┐
  email                         │
  hashed_password               │ FK (ON DELETE CASCADE)
  email_verified                │
  admin_approved                │
  is_active               verification_tokens
  role                      id (PK)
  created_at                user_id (FK) ──────────────┘
  updated_at                token
                            token_type  ['email_verify' | 'admin_approve']
                            expires_at
                            used_at
                            created_at
```

One user → many verification tokens (two per registration; more if re-sent).

## 6. Security Architecture

### 6.1 Password Security

| Concern | Mitigation |
|---|---|
| Plain-text storage | bcrypt with work factor ≥ 12 via `passlib[bcrypt]`. |
| Weak passwords | Enforce minimum 12 characters, mixed case, digit, and symbol at the Pydantic schema layer. |
| Timing attacks | `passlib` uses constant-time comparison to prevent hash inference. |

### 6.2 Token Security

| Concern | Mitigation |
|---|---|
| Token guessing | 64-byte URL-safe random string via `secrets.token_urlsafe(64)`. |
| Token reuse | `used_at` is set on first consumption; reuse is rejected. |
| Expiry | Tokens expire after 24 hours; checked against `expires_at` column. |

### 6.3 JWT Security

| Concern | Mitigation |
|---|---|
| Token forgery | Signed with HS256 using a strong server secret (32+ byte random key stored in env vars). |
| Token theft via XSS | Access token delivered via `HttpOnly` cookie; not accessible from JavaScript. |
| CSRF attacks | `SameSite=Lax` cookie attribute blocks cross-site form submissions. |
| Long-lived tokens | Short access token lifetime (1 hour). Refresh token pattern recommended for production. |

## 7. Technology Stack

| Layer | Library / Tool | Purpose |
|---|---|---|
| Web Framework | FastAPI | API routing, request validation, OpenAPI docs. |
| ORM | SQLAlchemy 2.x + asyncpg | Async database access and model definitions. |
| Migrations | Alembic | Schema version control and migration scripts. |
| Validation | Pydantic v2 | Request/response schema validation. |
| Password Hashing | `passlib[bcrypt]` | bcrypt password hashing with automatic salting. |
| JWT | `python-jose[cryptography]` | JWT creation, signing, and validation. |
| Email | FastAPI-Mail / smtplib | SMTP email dispatch for verification links. |
| Database | PostgreSQL 15+ | Primary relational data store. |
| Containerisation | Docker + docker-compose | Local development and deployment packaging. |

## 9. Open Questions & Decisions Required

| # | Question | Options | Recommendation |
|:---:|---|---|---|
| 1 | Should expired tokens be automatically cleaned up? | Scheduled cron job vs. lazy deletion on access. | Scheduled cleanup via `pg_cron` or APScheduler to keep the table lean. |
| 2 | How should token expiry failure be surfaced to the user? | Generic error page vs. re-send email button. | Provide a `/auth/resend-verification` endpoint with rate limiting. |
| 3 | Do we need refresh tokens for long sessions? | Short-lived JWT only vs. access + refresh token pair. | Implement refresh tokens stored in a separate HttpOnly cookie for production. |
| 4 | Single fixed admin email or multiple admin approvers? | Config-based single admin vs. admin role in `users` table. | Start with config-based; extend to role-based in v2. |
| 5 | Rate limiting on signup and login endpoints? | Application-level (`slowapi`) vs. infrastructure (API gateway / WAF). | Add `slowapi` as a quick win; migrate to gateway for production deployments. |
